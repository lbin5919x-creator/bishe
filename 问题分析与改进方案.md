# 交通信号控制系统问题分析与改进方案

## 一、核心问题总结

老师提出的三个关键问题：

1. **丁字路口和十字路口控制策略的差异体现不明显**
2. **单路口和多路口控制策略的差异体现不明显**
3. **固定时长和DQN的对比不公平（车流应该一致）**

## 二、问题根本原因分析

### 2.1 丁字路口 vs 十字路口差异不明显的原因

#### 当前实现问题：

**1. 状态空间设计相同**
- 丁字路口（T型）：3个方向（东、西、南），但状态提取方式与十字路口完全一致
- 十字路口（X型）：4个方向（东、西、南、北）
- 代码位置：[`environment/sumo_env.py:203-219`](environment/sumo_env.py:203)

```python
def _get_state(self) -> np.ndarray:
    """获取当前状态向量。"""
    state_features: List[float] = []
    # 1. 遍历所有受控车道，提取微观特征
    for lane in self.lanes:
        queue_length = traci.lane.getLastStepHaltingNumber(lane)
        occupancy = traci.lane.getLastStepOccupancy(lane)
        state_features.extend([queue_length, occupancy])
```

**问题**：没有针对路口拓扑结构的特殊处理，丁字路口缺少北向的特征应该被明确标识。

**2. 相位配置相同**
- 配置文件：[`config/settings.py:60-64`](config/settings.py:60)
```python
FIXED_TIME_PLANS = {
    "t_intersection": [40, 40],      # 2个相位
    "x_intersection": [30, 30, 30, 30],  # 4个相位
}
```

**问题**：
- 丁字路口只有2个相位（东西向、南向），配时各40秒
- 十字路口有4个相位，配时各30秒
- 但DQN智能体并没有针对不同相位数量进行特殊设计

**3. 车流模式差异不够显著**
- 丁字路口：[`scenarios/t_intersection/routes.rou.xml`](scenarios/t_intersection/routes.rou.xml)
  - 东西向：500 + 450 = 950 veh/h
  - 南向：300 + 300 = 600 veh/h
  
- 十字路口：[`scenarios/x_intersection/routes.rou.xml`](scenarios/x_intersection/routes.rou.xml)
  - 东西向：500 + 450 = 950 veh/h
  - 南北向：350 + 350 = 700 veh/h

**问题**：车流模式过于相似，没有体现丁字路口的非对称性特点。

### 2.2 单路口 vs 多路口差异不明显的原因

#### 当前实现问题：

**1. 多路口环境的协同机制不够强**
- 代码位置：[`environment/cascaded_env.py:460-479`](environment/cascaded_env.py:460)

```python
def _calculate_coordination_bonus(self) -> float:
    """全局协同奖励：惩罚所有连接路段的总排队积压。"""
    # ...
    normalized = total_link_queue / (cascaded_cfg.max_link_queue * len(self.link_edges))
    return -cascaded_cfg.link_queue_penalty * self.coordination_weight * normalized
```

**问题**：
- 协同权重 `coordination_weight=0.3` 较小
- 连接路排队惩罚系数 `link_queue_penalty=0.5` 也较小
- 导致多路口协同优势不明显

**2. 动作空间设计问题**
- 单路口：2个动作（保持/切换）
- 多路口（2个路口）：4个动作（2^2 = 4种组合）

**问题**：多路口的动作空间呈指数增长，但训练参数（episodes=200）相同，导致探索不充分。



### 2.3 固定时长 vs DQN 对比不公平的原因

#### 当前实现问题：

**1. 随机种子未统一**
- 评估脚本中seed默认为None
- 固定时长和DQN评估时车流完全不同
- 这是导致对比不公平的最关键原因

**2. 车流生成机制**
- SUMO使用vehsPerHour参数生成车辆
- 没有seed时，每次运行的车辆到达时间序列完全不同
- 固定时长控制器在某些随机序列下可能表现更好

**3. 评估回合数设置**
- 只评估20个回合，样本量较小
- 如果每个回合的seed不同，结果方差会很大
